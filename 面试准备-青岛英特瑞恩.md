### Q3：你项目里做的那些东西，跟 ERP 开发有什么关系？

ERP 核心就三件事：**数据怎么存、业务怎么流转、结果怎么呈现**。这三个我的项目全部覆盖了：

拿数据库设计来说——ERP 里最经典的模式是"主单据头 + 明细行"，比如一个采购订单有订单头（供应商、日期）和多条明细（物料、数量、单价），用订单号串联。我在项目里也是这种思路——`generation_logs` 一张表承载所有事件，用 `user_id` + `event_type` 串联整个用户行为链路，然后 Spark ETL 从这张表出发算出所有聚合数据。

再说报表——ERP 的看板本质上跟我的 9 类运营看板一样，都是预聚合 + 前端渲染。用 Spark 把原始数据算好写入聚合表，前端直接查，不用实时跑大查询。

所以虽然我做的项目业务场景跟制造业不一样，但**系统架构的骨架是通用的**。

## 二、项目深挖（5 题）—— 面试官会照着简历问

### Q4：简历写你设计了 19 张 MySQL 表，为什么要建这么多？不能几张表就搞定吗？

这个问题问得好。19 张表看起来多，本质上可以分为三层，每层解决不同的问题。

**第一层：流水层，就 1 张表——`generation_logs`。**

这是整个系统的唯一数据入口。用户每次生成内容、下载图片、提交评分，全部写进这张表。为什么要用一张表而不是分开？因为用户的行为是连续的——他生成完可能下载，下载完可能评分。如果拆成三张表，后面做分析时要 JOIN，数据量上来之后 JOIN 会非常慢。一张表、用一个 `event_type` 字段区分事件类型，查询时 WHERE 一下就行，不用 JOIN。

**第二层：聚合层，大概十来张表——比如 `user_profiles`、`style_popularity_30days`。**

这块是最关键的。我举个例子——运营看板上有一个"最近 30 天风格趋势"的折线图，要展示每种风格每天的使用次数变化。如果每次打开看板都从 `generation_logs` 原始表实时 GROUP BY 日期和风格，表里两万多条数据还好，但如果数据量大到几十万、上百万，这个查询就要秒级别了，而且多人同时打开看板会对数据库造成很大压力。

所以我的做法是——用 PySpark 每天晚上跑一次离线 ETL，把趋势、热度、评分分布这些算好，写进聚合表。前端打开看板直接 `SELECT * FROM style_trend_30days`，毫秒级返回。

这就是为什么表多——**本质上是用空间换时间，用预计算换查询性能。**

**第三层：推荐层，2 张表——`user_style_recommendations`、`user_keyword_preferences`。**

推荐算法（ALS 协同过滤）跑一次很耗时，不可能用户每次刷新都重新训练。所以训练结果写进表里，前端直接查。这也是预计算的思想。

总结一下就是：**流水表保证数据完整，聚合表保证查询快，推荐表保证体验好。** 各有各的用途，不是故意堆数量。

### Q5：你简历写了"双数据源架构，三级查询降级"，为什么又要 MySQL 又要 Hive？降级怎么做的？

这是整个项目里我觉得设计得比较好的一点。

**为什么两个数据源？**

MySQL 和 Hive 的定位不一样。MySQL 存的是我 Spark ETL 跑出来的聚合结果——数据量小、查询快，适合前端实时请求。Hive 存的是原始日志数据——数据量大、查询慢，但能做一些 MySQL 不方便做的全量分析。

举个例子：运营想查"上个月所有用户的平均生成耗时"，如果聚合表里有直接取就行。如果没有，从 Hive 查原始日志现算——虽然慢一点但起码能算出来。

**三级降级怎么做的？**

我代码里的逻辑很简单：

```
第一级：查 MySQL 聚合表
   ↓ 如果 MySQL 挂了 / 返回空
第二级：查 Hive 原始表现算
   ↓ 如果 Hive 也挂了
第三级：返回前端一套模拟数据，保证页面不白屏
```

实现上就是每个数据源写一个 service 类，查询方法失败返回 None，API 层用 if-else 串联：MySQL 有结果就用，没有就试 Hive，还没有就用兜底数据。

这个设计的目的就一个——**保证前端任何时候打开都有东西看**，而不是报错或者转圈。

### Q6：你写了"解决 MySQL 连接池耗尽与 Broken pipe 异常"，说说怎么回事？

这两个问题是我在 Linux 部署之后遇到的真实坑。

**连接池耗尽**——Flask 默认是单线程多进程或者用 gevent 协程，我用的是标准模式，每次请求过来都尝试从连接池拿一个连接。问题是长时间运行后，有些连接已经失效了但没被回收，新请求来的时候拿不到有效连接，就报"连接池耗尽"。

我的解决办法比较直接——**干脆不用连接池**。每次执行 SQL 之前新建一个连接，执行完立刻关闭。MySQL 建立连接其实很快，几毫秒的事，项目初期并发量不大，这个方案是最稳的。以后如果遇到高并发瓶颈，再考虑用连接池 + 健康检查的方案。

**Broken pipe**——这个更常见。MySQL 服务端默认有一个 `wait_timeout` 参数，8 小时。如果客户端拿了一个连接，闲置超过 8 小时没用，MySQL 服务端就单方面断开了。但客户端不知道，下次用这个连接发 SQL 的时候就报 "Broken pipe"。

我的处理是加了三层超时——`connect_timeout` 10 秒、`read_timeout` 30 秒、`write_timeout` 30 秒，再配合失败重试：如果执行 SQL 的时候报错（包括 Broken pipe），自动重试一次，中间间隔 1 秒。

这个思路其实跟 PostgreSQL 也是一样的——这类问题不是 MySQL 独有的，任何数据库连接都有超时断连的风险。

### Q7：阿里云大模型异步任务轮询和重试怎么设计的？

这个功能是用户点"生成"按钮之后，后端调用阿里云通义万相生成图片。

流程是这样的：先调通义万相的 API 提交生成任务，它不是同步返回结果，而是返回一个 `task_id`。然后我拿着这个 task_id 去轮询任务状态。

具体设计上我定了三个参数：最多轮询 12 次，每次间隔 5 秒，也就是总共等 60 秒。每次轮询查到的状态有三种情况：

- `SUCCEEDED`——任务完成，取出图片 URL 返回给前端
- `FAILED` 或 `CANCELED`——任务失败，直接返回一张降级占位图
- 其他状态——还没完成，等 5 秒继续轮询

如果 12 次轮询完还没完成，那就超时了，一样返回降级图。

**为什么要有降级？** 因为用户体验不能断——一个用户点生成等了一分钟结果给我报错，下次他就不用了。降级图至少让他看到个结果，知道系统是活的。

**如果让我重新设计**，我会把轮询改成回调模式——提交任务时传一个 webhook 地址，任务完成后阿里云主动通知我的后端，不用自己一直查。但通义万相当时没提供这个能力，所以只能用轮询。

### Q8：你说用 PySpark 处理了两万多条数据，构建了用户分层模型——具体做了什么？

这个是整个数据分析链路的核心。我分几步讲。

**数据来源**：三路数据合在一起——我自己生成的模拟数据、系统实时日志、还有一个 Flickr30K 的公开图像描述数据集。用 PySpark 的 `unionByName` 把三路数据合并成一张统一格式的大表，总共两万多条。

**数据增强**：原始数据有些字段是空的。比如用户年龄、性别、评分——不是每条记录都有。我用 Spark 做了缺失值填充，有真实数据就用真实的，没有就用随机值补上。另外还算了几个衍生字段：prompt 长度、内容质量评级（高中低）。

**特征工程**：把文字变成算法能理解的数字。中文提示词先分词再过滤停用词；风格（比如"赛博朋克"、"水彩画风"）用 StringIndexer 编码成数字；时间戳拆成小时、星期几、是否周末。

**用户聚类**：每个用户提取了 8 个维度的特征——生成次数、平均耗时、内容长度、评分、关键词量、风格多样性、下载量、活跃时段。用 KMeans 聚成 4 类：

- **高活跃创作型**：生成频繁、内容质量高、风格多样
- **普通体验型**：偶尔用、各方面指标中规中矩
- **高质量偏好型**：用得不多但评分高、对内容质量有要求
- **低频探索型**：注册了基本没用过

**推荐算法**：按这 4 类分别训练 ALS 协同过滤模型。为什么按分类分别训练而不是全局训练？因为同类用户的口味更接近——你拿"低频探索型"的用户去推荐"高活跃创作型"偏好的风格，效果肯定不好。分群训练让推荐更精准。

**落地**：聚类结果和推荐结果都写进 MySQL 表，前端看板和推荐接口直接查。

## 三、技术迁移（2 题）

### Q9：你之前用 MySQL，这个岗位用 PostgreSQL，你怎么看？

关系型数据库的**核心是一样的**——SQL 标准、表设计范式、索引优化思路都是通用的。差别在细节上。

PostgreSQL 有几个比 MySQL 强的地方：第一是数据类型丰富，比如 JSONB 可以直接存 JSON 并且支持索引查询。我项目里 keyword_analysis 表有个字段存 JSON 趋势数据，是用 VARCHAR 存的，每次读写都要手动解析。换成 PostgreSQL 直接用 JSONB，省事而且查询更快。第二是并发控制更强，PostgreSQL 用 MVCC，读写互不阻塞，对 ERP 这种并发写入多的场景更友好。

迁移的话思路也简单：表结构映射（DATETIME → TIMESTAMP 之类的）、连接驱动从 pymysql 换成 psycopg2、存储过程如果有的话需要重写（语法有差异）。工作量不算大。

### Q10：如果让你设计一个简单 ERP 的表结构，你会怎么做？

我会从"物料"和"单据"两条主线出发。

主数据层先有 `item_material`（物料主表），存物料编码、名称、单位、规格这些基础信息。如果产品有层级结构，再加一个 `bom`（物料清单表），记录哪个物料由哪些子物料组成、各用多少。

业务层用"头+行"模式——比如一张采购订单，`purchase_header` 存订单号、供应商、日期、状态；`purchase_line` 存订单号、物料、数量、单价。两张表用订单号关联。工单也一样，`work_header` + `work_line`。

库存层单独一张 `inventory`，物料 + 仓库 + 当前库存量，采购入库就加，工单领料就减。

这个设计思路跟我项目里 `generation_logs` 是一样的——用唯一标识（订单号 / 用户ID）串联业务流程，流水表和聚合表分层处理。具体业务字段不同，但骨架通用。
