### 1.2 JWT 鉴权实现

**代码位置**：`backend/app.py` 第 15-80 行

```python
JWT_SECRET = "your-super-secret-jwt-key-change-in-production"
JWT_ALGORITHM = "HS256"

def authenticate_user():
    auth_header = request.headers.get('Authorization')
    if not auth_header or not auth_header.startswith('Bearer '):
        return None
    token = auth_header.replace('Bearer ', '')
    payload = jwt.decode(token, JWT_SECRET, algorithms=[JWT_ALGORITHM])
    return payload
```

**设计要点**：

- 登录成功后生成 Token，有效期 1 天（`exp` 字段），包含 `user_id`、`username`、`role` 三个关键信息
- 需要登录的接口在函数开头调用 `authenticate_user()`，返回 None 则直接返回 401
- 需要管理员权限的接口（运营数据）额外检查 `role == 'admin'`，不满足返回 403
- 不用 Flask 中间件装饰器，而是在每个接口函数体内显式调用——好处是鉴权逻辑一目了然，不会因为装饰器顺序问题导致鉴权遗漏

**Token 生成**（`/api/login` 第 222-228 行）：

```python
token_payload = {
    'user_id': user['user_id'],
    'username': user['username'],
    'role': user['role'],
    'exp': datetime.utcnow() + timedelta(days=1)
}
token = jwt.encode(token_payload, JWT_SECRET, algorithm=JWT_ALGORITHM)
```

前端拿到 Token 后存入 localStorage，后续请求统一在 Header 里带 `Authorization: Bearer <token>`。

### 1.3 19 张 MySQL 业务表设计

表不是随意堆出来的，而是按照"**数据从哪来 → 怎么算 → 怎么用**"这条链路分层设计的。

#### 第一层：业务流水层（1 张）

**`generation_logs`** — 整个系统的唯一数据入口。

所有的用户行为（生成内容、下载图片、提交评分）全部写入这一张表，用 `event_type` 字段区分：

- `generate`：生成事件（提示词、风格、耗时、内容长度）
- `download`：下载事件
- `rating`：评分事件（1-5 分）

**为什么只建一张表而不是三张？**

如果把生成、下载、评分拆成三张表，后续做用户分析时需要 JOIN。以 `user_id` 为主键 JOIN 三个表，数据量大之后会非常慢。一张表 + `event_type` 区分，查询时 `WHERE event_type = 'generate'` 就行，零 JOIN。

**代码位置**：数据写入见 `backend/app.py` 第 412-435 行（`INSERT INTO generation_logs`），写入时机是每次 `/api/generate` 被调用时；批量写入见 `scripts/spark_etl.py` 第 911-932 行（Spark 写 MySQL）。

---

#### 第二层：预聚合层（约 11 张）

这一层的核心设计思想是 **预计算替代实时查询**。

举个例子：运营看板上有一个"最近 30 天风格趋势"的折线图，需要展示每种风格每天的生成次数。如果每次打开看板都从 `generation_logs` 实时查：

```sql
SELECT DATE(timestamp) as date, style, COUNT(*) 
FROM generation_logs 
WHERE timestamp >= DATE_SUB(NOW(), INTERVAL 30 DAY)
GROUP BY date, style
ORDER BY date
```

当数据量达到数万甚至数十万条时，这个查询需要全表扫描 30 天的数据再聚合，响应时间轻松上秒级。如果多个运营同时打开看板，数据库压力会很大。

**我的方案**：用 PySpark 每天跑一次离线 ETL，把聚合结果预先算好，写入专门的聚合表。前端打开看板时直接：

```sql
SELECT * FROM style_trend_30days ORDER BY stat_date
```

毫秒级返回，数据库零压力。

这个思路在项目中复用了很多次，下面是所有聚合表及其用途：

| 表名 | 对应的看板 | 聚合维度 |
|------|----------|---------|
| `user_age_distribution` | 用户年龄分布饼图 | 按年龄段分组（0=未知, 1=<18, 2=18-24, ..., 7=50+） |
| `user_gender_distribution` | 用户性别分布饼图 | 按性别分组（0=女, 1=男, 2=未知） |
| `user_active_period_distribution` | 活跃时段分布图 | 按时段分组（0=深夜, 1=上午, 2=下午, 3=晚间） |
| `user_behavior_7days` | 7 天用户行为趋势图 | 按天聚合（生成数、下载数、活跃用户数） |
| `style_popularity_30days` | 风格热度排行 TOP10 | 按风格聚合（使用次数、下载量、评分、独立用户数） |
| `style_trend_30days` | 30 天风格趋势折线图 | 按天+风格双维度聚合 |
| `rating_distribution` | 用户满意度评分分布 | 按评分值（1-5）聚合 |
| `keyword_analysis` | 热门关键词词云 | 按关键词聚合（使用频次、热度分、趋势 JSON） |
| `generation_efficiency` | 生成效率分布图 | 按耗时区间聚合（0-5s/5-10s/10-30s/30-60s/60s+） |
| `user_profiles` | 用户画像总览 | 按用户聚合（总次数、平均耗时、平均评分、偏好风格、所属群体） |
| `user_active_period_distribution` | 活跃时段分析 | 按活跃时间段聚合 |

**代码位置**：所有聚合表由 `scripts/spark_etl.py` 第 894-1053 行的 `save_results_to_mysql()` 函数通过 Spark DataFrame 的 `.write.jdbc()` 写入 MySQL。

#### 第三层：推荐结果层（2 张）

推荐算法（ALS 协同过滤）一次训练耗时较长，不可能每次用户刷新页面都重跑。所以训练结果预先算好存表：

**关于"没有建表语句"的问题**：

### 1.4 MySQL 连接池耗尽与 Broken pipe 异常

**代码位置**：`backend/services/mysql_service.py` 第 67-137 行

#### 连接池耗尽

**现象**：系统运行一段时间后报"连接池耗尽"，新请求拿不到连接。

**原因**：Flask 应用启动时创建的数据库连接，长时间闲置后有些已失效但未被回收，可用连接越来越少。

**解决方案**：**放弃连接池模式，改为"即建即用即关"**。

```python
def execute_query(self, query, params=None, max_retries=2):
    for attempt in range(max_retries + 1):
        try:
            # 每次都创建新连接
            connection = pymysql.connect(
                host=self.host, port=self.port,
                user=self.username, password=self.password,
                database=self.database, charset='utf8mb4',
                cursorclass=pymysql.cursors.DictCursor,
                autocommit=True,
                connect_timeout=10, read_timeout=30, write_timeout=30,
            )
            with connection.cursor() as cursor:
                cursor.execute(query, params)
                result = cursor.fetchall() if query.strip().upper().startswith('SELECT') else cursor.rowcount
            connection.close()  # 执行完立即关闭
            return result
        except ...:
            ...
```

MySQL 建立连接只需要几毫秒，项目初期并发量不大，这个方案的性能损耗完全可以接受，但稳定性大幅提升。

#### Broken pipe

**现象**：MySQL 报错 "MySQL server has gone away" 或 "Broken pipe"。

**原因**：MySQL 服务端默认 `wait_timeout = 8小时`，连接闲置超过 8 小时后服务端主动断开，但客户端不知情，下次发 SQL 时才发现连接已断。

**解决方案**：三重超时保护 + 失败自动重试。

```python
# 三重超时
connect_timeout=10,    # 建立连接超时 10 秒
read_timeout=30,       # 读取超时 30 秒
write_timeout=30,      # 写入超时 30 秒
```

```python
# 失败重试（最多 2 次，间隔 1 秒）
except (pymysql.err.OperationalError, pymysql.err.InterfaceError) as e:
    if attempt < max_retries:
        time.sleep(1)
        continue
```

这个解决思路对 PostgreSQL 同样适用——数据库连接都有超时断连风险，核心解法都是"超时控制 + 重试机制"。

## 二、双数据源设计与数据处理

### 2.1 MySQL + Hive 双数据源架构

**为什么需要两个数据源？**

MySQL 和 Hive 定位不同：

| | MySQL | Hive |
|---|-------|------|
| 存储内容 | 预聚合结果（数据量小） | 原始日志（数据量大） |
| 查询速度 | 毫秒级 | 秒级（底层是 MapReduce） |
| 适合场景 | 前端实时请求（看板） | 全量数据分析和 Ad-hoc 查询 |
| 数据来源 | Spark ETL 写入 | 应用日志 + Flickr30K 数据集 |

**代码结构**：

```
backend/services/
├── mysql_service.py   # MySQL 服务类（查聚合表）
├── hive_service.py    # Hive 服务类（查原始日志）
└── aigc_service.py    # 大模型服务类
```

每个 service 独立封装连接管理和查询逻辑，API 层（`app.py`）按优先级调用。

## 三、大模型集成与可视化看板

### 3.1 阿里云大模型异步任务轮询与重试

**代码位置**：`backend/services/aigc_service.py`

#### 整体流程

用户点"生成" → 后端调用两个模型：

1. **通义千问（qwen-turbo）** → 生成文案（标题 + 正文），同步返回，耗时约 2-5 秒
2. **通义万相（wanx-v1）** → 生成图片，异步任务，提交后拿 `task_id`，**轮询等待**结果

#### 异步轮询设计

```python
def wait_for_image_task(self, task_id, max_attempts=12, interval=5):
    """轮询图片生成任务，最多等 60 秒"""
    for attempt in range(max_attempts):
        response = requests.get(f"{self.task_query_url}{task_id}", ...)
        task_status = result.get('output', {}).get('task_status')

        if task_status == 'SUCCEEDED':
            return result['output']['results'][0]['url']   # 成功，返回图片 URL
        elif task_status in ['FAILED', 'CANCELED']:
            return self.get_fallback_image_url(...)         # 失败，降级图
        else:
            time.sleep(interval)                            # 还没完成，等 5 秒继续
```

**三个设计参数**：

- `max_attempts = 12`：最多轮询 12 次
- `interval = 5`：每次间隔 5 秒
- **总超时 = 60 秒**（12 × 5）

超过 60 秒未完成 **不是报错给前端**，而是返回一张降级占位图。用户看到一个结果（即使不是 AI 生成的），比看着报错信息好得多。

#### 降级设计

整个 AIGC 链路有三处降级：

1. **API Key 未配置** → 直接返回模拟文案和占位图
2. **API 调用失败** → 返回预设的文案模板（根据 prompt 和 style 拼一句）
3. **图片任务超时/失败** → 返回 `/static/fallback/` 下的占位图

**代码中的降级函数**：

```python
def get_fallback_text_content(self, prompt, style):
    title = f"{prompt}的{style}之旅"[:10]
    content = f"在{style}的风格下，{prompt}展现出独特的魅力。..."
    return title, content

def get_fallback_image_url(self, prompt, style):
    timestamp = datetime.now().strftime("%Y%m%d%H%M%S")
    return f"/static/fallback/{prompt}_{style}_{timestamp}.jpg"
```

**支撑多维度数据统计的方式**：

- 运营端支持**日期范围筛选**（`start_date` / `end_date` 参数），前端 Vue 组件传递日期参数，后端 SQL 加 `WHERE stat_date BETWEEN %s AND %s`
- 用户画像看板中所有图表数据是一次 API 请求全部返回的（`/api/dashboard/user-profile` 一个接口返回所有 9 类数据），而不是 9 次请求——减少网络开销

## 四、项目架构全景图

```
用户浏览器（Vue.js）
    │
    ▼
Flask 后端（backend/app.py）
    │
    ├── /api/login ────────── JWT 鉴权
    ├── /api/generate ─────── 阿里云大模型（aigc_service.py）
    ├── /api/dashboard/* ──── MySQL（聚合数据）→ Hive（降级）→ 兜底数据
    └── /api/recommendations ─ MySQL / Hive（推荐结果）
    │
    ▼
数据层
    ├── MySQL（aigc_platform 库，19 张表）
    │   ├── 流水层：generation_logs
    │   ├── 聚合层：user_profiles, style_trend_30days, ...
    │   └── 推荐层：user_style_recommendations, user_keyword_preferences
    │
    └── Hive（aigc_platform 库，2 张表有 DDL）
        ├── user_profiles
        └── generation_logs
    │
    ▼
离线处理（定时任务）
    └── PySpark ETL（scripts/spark_etl.py）
        ├── 三数据源合并（模拟 + 实时日志 + Flickr30K）
        ├── 特征工程 + KMeans 聚类
        ├── ALS 推荐训练
        └── 结果写入 MySQL
```
